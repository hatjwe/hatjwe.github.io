    k8s日志审计调研

    k8s日志审计k8s组件在容器编排创建删除等过程中都会将行为记录为日志，可以通过日志审计发现某些黑客的敏感行为进行预警

    方案详情

    目前主要形成以下方案，详情及优缺点提供参考（主要推荐:监听收集集群日志+脚本规则处理）

    侵入性 优先级
    监听收集集群日志+脚本规则处理 无 2
    WEBhook，apiserver行为验证 3
    kube-apiserver配置yaml策略规则 策略配置需要修改配置文件 1

    kube-apiserver配置yaml策略规则

    审计政策定义了关于应记录哪些事件以及应包含哪些数据的规则。处理事件时，将按顺序与规则列表进行比较。第一个匹配规则设置事件的 [审计级别][auditing-level]。已知的审计级别有：
    None -符合这条规则的日志将不会记录。
    Metadata -记录请求的 metadata（请求的用户、timestamp、resource、verb 等等），但是不记录请求或者响应的消息体。
    Request -记录事件的 metadata 和请求的消息体，但是不记录响应的消息体。这不适用于非资源类型的请求。
    RequestResponse -记录事件的 metadata，请求和响应的消息体。这不适用于非资源类型的请求。
    您可以使用 –audit-policy-file 标志将包含策略的文件传递给 kube-apiserver。如果不设置该标志，则不记录事件。注意 rules 字段必须在审计策略文件中提供。
    这种方式为k8s自带功能策略方面比较丰富，缺点在于需要配置apiserver，但apiserver多以system pod形式启动因此会导致k8s重启。
    优点：官方功能获取日志都很方便
    缺点：需要配置apiserver参数，大程度影响k8s运行，消耗性能较高开启后部署资源有明显卡顿

    监听收集集群日志+脚本规则处理

    获取方式：

    kubectl cluster-info dump 获取所有pod yaml信息,kube-apiserver,kube-controller,kube-proxy,kube-scheduler组件日志（socket形式，外部工具）

    数据结构

    监听收集集群日志

    journalctl -l -u kubelet (流文件)
    短时间间隔不断监听我们可以获取k8s集群日志的信息，包括pod创建时用yaml，server，namespace,ClusterRole,容器日志，事件日志

    处理分析数据

    编写脚本主要是将获取到的数据有一些包括pod的创建节点树，events事件，服务等进行相关联，通过观察日志变化获取行为变化对特殊行为进行预警

    工作流程

    运行监听功能每隔一段时间获取日志，获取日志日志json格式输出–json字段解析抽象化当前k8s集群状态–加载规则判断逻辑–根据时间间隔观察是否触发危险行为产生警告.（日志分发）

    优点：无侵入性只是获取k8s所有组件日志，之后通过脚本进行数据加载，集群状态模拟，通过时间对比行为变化进行预测，
    缺点：通过观察k8s日志发现比较粗略更像是程序的报错，因此对于一些信息的获取无法从日志直观看出比如
    1.创建secert
    没有在日志中进行体现（创建secert获取不到)。

    WEBhook apiserver行为消息+规则引擎（最终选择方案）

    K8S中提供了自定义资源类型和自定义控制器来扩展功能，还提供了动态准入控制，其实就是通过Webhook来实现准入控制，分为两种：验证性质的准入 Webhook （Validating Admission Webhook） 和 修改性质的准入 Webhook （Mutating Admission Webhook）

    创建RBAC—证书认证—运行webhook
    优点:在apiserver行为处获取所有行为，直接进行规则验证，比较灵活方便
    缺点：会对api响应有一定影响。

    缺点：会对api响应有一定影响，在master节点部署
